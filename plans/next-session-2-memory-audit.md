# Сессия: Memory Audit — что работает, что лишнее

## Context

MetaMode v1 — persistent wiki-memory поверх Claude Code. Пользователь — начинающий разработчик, боится что-то сломать. Нужно разобраться в текущей ситуации с памятью: какие системы работают, какие лишние, нет ли конфликтов.

Прочитай `CLAUDE.md` в корне проекта для полного контекста архитектуры.

## Проблема

Сейчас у пользователя работают **три параллельные системы памяти**, и непонятно, как они соотносятся:

### 1. MetaMode Wiki (Karpathy method) — наша система
- **Где:** `C:\Users\Voland\Dev\MetaMode\knowledge/`
- **Как работает:** hooks → flush.py → daily logs → compile.py → wiki-статьи с wikilinks
- **Inject:** SessionStart hook читает `index.md` + recent log и инжектит в контекст
- **Качество:** human-in-the-loop (pending review), structured articles
- **Статус:** полностью реализована, Phase B+C завершены

### 2. Mem0 MCP Plugin (cloud) 
- **Где:** `~/.claude/plugins/mem0/` (MCP сервер + UserPromptSubmit hook)
- **Как работает:** MCP tools (`search_memories`, `add_memory`, etc.) через Mem0 cloud API
- **Inject:** SessionStart hook от плагина печатает bootstrap prompt ("call search_memories")
- **Качество:** автоматическое извлечение через Mem0 LLM, ранее оценивалось как "97% junk"
- **Решение от 2026-04-11:** "Mem0 cloud — drop'нут" (см. `project_final_decision.md`)
- **НО:** плагин всё ещё установлен и активен! MEM0_API_KEY в env vars.

### 3. Claude Code Native Auto-Memory
- **Где:** `C:\Users\Voland\.claude\projects\c--Users-Voland-Dev-MetaMode\memory\`
- **Как работает:** Claude Code сам пишет файлы в `memory/` + `MEMORY.md` индекс
- **Inject:** автоматически загружается при старте сессии в этом проекте
- **Качество:** зависит от того, что Claude решит сохранить
- **Статус:** активно используется (22 файла, в основном project context)

## Задачи аудита

### 1. Инвентаризация: что реально активно

Проверь каждую систему:

**Wiki:**
- Прочитай `knowledge/index.md` — сколько статей? О чём?
- Проверь `daily/` — сколько логов? Последняя дата?
- Есть ли вообще полезный контент, или только тестовые данные?

**Mem0:**
- Вызови `search_memories` с query "MetaMode" — что возвращает?
- Вызови `get_memories` — сколько записей, какого качества?
- Проверь `~/.claude/plugins/mem0/` — что там лежит, как настроен hook

**CC Auto-Memory:**
- Прочитай `MEMORY.md` — что там сохранено?
- Какие файлы в `memory/` ещё актуальны, а какие устарели?

### 2. Анализ конфликтов

- Не дублируют ли системы друг друга?
- SessionStart: сколько inject'ов происходит? (wiki hook + mem0 hook + CC auto-memory = тройной контекст?)
- Не раздувается ли контекст при старте из-за трёх систем?

### 3. Рекомендация

На основе аудита дать чёткую рекомендацию:

**Вариант A: Wiki only**
- Удалить mem0 плагин, очистить env vars
- CC auto-memory оставить (не мешает, полезна для project context)
- Wiki — единственный источник правды для знаний

**Вариант B: Wiki + CC auto-memory (разделение ролей)**
- Wiki = долгосрочные знания (концепции, паттерны, уроки)
- CC auto-memory = проектный контекст (фазы, решения, прогресс)
- Mem0 = удалить

**Вариант C: Оставить всё как есть**
- Если mem0 не мешает и не стоит денег — зачем удалять?
- Просто не использовать активно

**Вариант D: Что-то другое**
- Может есть вариант лучше?

### 4. Если решили убрать mem0

Пошаговая инструкция:
1. Сохранить полезные записи из mem0 (если есть) в wiki RAW inbox
2. Удалить плагин: как?
3. Убрать MEM0_API_KEY из env vars
4. Убрать bootstrap prompt из SessionStart (если он от mem0)
5. Проверить что ничего не сломалось

### 5. Если решили оставить CC auto-memory

Разобраться:
- Файлы в `memory/` — какие актуальны, какие устарели?
- `MEMORY.md` — нужно ли почистить?
- Не дублирует ли CC auto-memory информацию из wiki?
- Как разграничить: что сохранять в wiki, что в CC memory?

## Constraints

- Не ломать то, что работает
- Не удалять данные без подтверждения пользователя
- Объяснять каждое решение: почему именно так
- Пользователь — начинающий разработчик, объяснять простым языком

## Ожидаемый результат

1. Таблица: система → статус → полезность → рекомендация
2. Чёткое решение: что оставить, что убрать

4. Если оставляем — правила разграничения (что куда сохранять)
