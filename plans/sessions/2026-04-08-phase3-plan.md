## Ключевые решения

- **Q1 metadata schema.** `{project, cwd, session_id, timestamp}` + `user_id="voland"` отдельным параметром. **Why:** `cwd` — надёжный дискриминатор проекта, остальное — для отладки и будущих TTL. Mem0 metadata schemaless, расширяем без миграции.
- **Q2 hook contract — snap-решение.** Claude Code сам передаёт `session_id` и `transcript_path` в stdin обоих хуков. Никакого state-файла / env-переменной / Mem0-метадаты для синхронизации. **Why:** это штатный механизм Claude Code, найден в hooks docs — незачем изобретать свой.
- **Q3 summary capture.** Stop hook читает `transcript_path` (JSONL), парсит messages, передаёт сырыми в `mem0.add(messages=...)`. Mem0 сам извлекает facts. **Why:** это штатный путь Mem0 SDK, альтернативы (просить агента саммаризировать / свой summarizer) — хрупче и дороже.
- **Q4 retrieve параметры.** Стартовые `top_k=10`, фильтр по `user_id + metadata.cwd`. **Why:** success criteria фиксируют качество recall, не числа — эмпирически подстроим в Phase 4.
- **Архитектура plugin-а — 5 файлов.** `plugin.json` + 2 hook-скрипта + `mem0_client.py` (вся логика) + тесты. Вся чистая логика в `mem0_client.py` (тестируется), hooks — тонкие stdin/stdout адаптеры. **Why:** тестируемость + удержаться в границах pitch-формата.
- **Error handling: hook никогда не блокирует.** Любая ошибка → `error.log` + exit 0. **Why:** secondary success criterion говорит, что нестабильность hooks = tech debt, не провал гипотезы; primary — ценность recall, и пропущенный recall лучше сломанной сессии.
- **ADR-0002 не создаётся.** Все четыре решения обратимы за ≤1 сессии, триггер §3 foundation-template (≥2 альтернатив И сложно откатить И "почему через 2 недели") не срабатывает ни для одного.
- **5 задач = appetite.** T1 skeleton → T2 core logic + tests → T3 Stop hook → T4 SessionStart hook → T5 реальное тестирование. Порядок: сначала write path (T3), потом read path (T4), потому что recall без накопленных записей проверять нечем.
