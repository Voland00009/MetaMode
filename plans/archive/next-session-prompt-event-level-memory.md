## Твоя первая задача в этой сессии

Пользователь прямо сказал: **«Я не совсем понял, что такое event-level память как potential moat. Попробуй в новой сессии сначала рассказать, что это.»**

Это блокирующий пункт. Прежде чем двигаться в Фазу A (инвентарь инструментов), нужно разобрать концепцию **event-level memory vs session-level memory**, почему она может быть конкурентным преимуществом против будущей встроенной памяти Anthropic, и что именно в Claude Code 2026 даёт возможность её реализовать.

**Структура объяснения, которую ждёт пользователь:**

1. **Что такое session-level память** (то, что делают Mem0, Karpathy, claude-mem, и то, что почти наверняка сделает Anthropic) — запись на выходе из сессии одним большим куском. Один Stop hook → один summary → одна запись. Гранулярность = сессия.
2. **Что такое event-level память** — запись на каждое значимое событие внутри сессии. Claude Code 2026 выставляет хуки `PostToolUse`, `PreToolUse`, `UserPromptSubmit`, `Notification`, `SubagentStop`. Каждый из них — это отдельная точка, где мы узнаём что-то конкретное: «агент запустил `pytest` и получил pass», «юзер ответил "нет, не так"», «агент попробовал rm и получил permission denied», «сабагент вернулся с результатом X». Гранулярность = одно решение / один факт.
3. **Почему это potentially moat:**
   - Anthropic почти наверняка выкатит session-level память (это дефолт, это соответствует тому, как они думают про Projects и Artifacts). Быть просто «ещё одним Mem0» поверх их решения — проиграть в первую неделю.
   - Event-level даёт качественно другую ценность: self-learning из **действий**, а не из **разговоров**. Разница как между «я тебе рассказал, что мы починили баг» и «я видел, что ты запустил команду X, она провалилась, ты запустил Y, она прошла — запомнил связку».
   - Никто из публичных решений этого не делает, потому что это требует стриминг-processing (каждое событие — отдельное обращение) и хорошей схемы, чтобы не захлебнуться в шуме. Это **инженерный порог входа**, а не очевидный паттерн. Именно это делает его potential moat.
4. **Риски и unknown-unknowns:**
   - Шум vs сигнал. 95% PostToolUse событий — не уроки, а просто Read/Grep. Нужен фильтр (какие события вообще стоят записи).
   - Стоимость. Если писать в мем-провайдер на каждое событие — бюджет и latency улетят. Значит — writer batching или локальное накопление.
   - Privacy. PostToolUse payload может содержать содержимое файлов, в том числе секреты.
   - Запись ≠ знание. Сырые события бесполезны без компиляции (это как раз то, зачем Карпати ввёл compile-этап).
5. **Какая картинка получается:**
   - Stream: хуки пишут сырые события в локальный append-only лог (markdown или jsonl — решим в Фазе C).
   - Compile: отдельный процесс (scheduled через CronCreate, или on-demand через SessionStart тригер по времени, или background через Agent SDK) периодически перерабатывает лог в переиспользуемые «уроки».
   - Retrieve: lazy — SessionStart грузит только index, конкретные уроки подкачиваются скиллом по триггеру или через UserPromptSubmit hook-роутером.

После этого объяснения — **чекин с пользователем:** поймал ли он идею, хочет ли идти дальше, или нужно подрегулировать. И только потом — Фаза A.

### 6 архитектурных дыр в текущем Phase 1, которые я увидел

1. **Жадный SessionStart.** Q4 жёстко фиксирует `top_k=10 + фильтр cwd + контекстная строка`. Это жадная загрузка на каждом старте — противоречит требованию №3 (токен-экономия).
2. **Summary делает Mem0 своим LLM** (`infer=True`). Мы платим вендору за извлечение фактов, когда у нас уже есть подписка на Claude. И не контролируем схему извлечения — нарушение требования №5 (прозрачность) и частично №6 (moat).
3. **Нет PreCompact хука.** Длинная сессия с auto-compact теряет ранние решения до Stop. У Карпати закрыто.
4. **Self-learning явно отложен** в pitch в Phase 2 («failure-tagging — откладывается»). Но self-learning — это требование №2, главный moat. Значит Phase 1 проверяет не ту гипотезу, которую надо проверить.
5. **Нет компиляции.** Mem0 extract facts ≠ compile логов в переиспользуемые статьи. У Карпати есть compile, у нас аналога нет.
6. **Нет cross-project мостика.** Жёсткий фильтр по `metadata.cwd` убивает сценарий «решил эту проблему в проекте А, теперь делаю то же в проекте Б». Карпати решает через путь в CLAUDE.md.

### Что реально ценно у Карпати (не симпатичные ритуалы)

- **Компилятор как отдельный процесс.** Запись ≠ знание. Между ними LLM-переработка. Это фундамент self-learning.
- **Compile-on-idle через Claude Agent SDK на подписке.** Буквально нулевой дополнительный расход, только токены существующей подписки. Закрывает требование №4 (cost <50/mo).
- **Линтер как health-check.** Находит битые ссылки, orphan-статьи, устаревание. Не ритуал — защита от деградации базы.
- **`index.md` как навигационная точка входа.** LLM получает ~200 слов индекса, сама решает, куда идти, тянет только нужное. Лучший token budget: модель делает то, что embedding-search делает с эрозией точности.

### Что в Mem0 всё равно имеет смысл

Не целиком хранилище, а **семантический поиск поверх markdown-базы** — акселератор, не единственный слой. «Либо Mem0, либо Obsidian» — ложная дихотомия.

### Что в `AI Memory System` (старая идея в input/) НЕ годится

Tool-driven: `/reflect`, `/lessons`, `/kb-maintain` — всё руками. Это та самая «провал гипотезы, если нужно вручную».

### Вторая важная мысль (кроме event-level): скиллы как механизм lazy retrieval

Скилл активируется по триггеру, а не грузится сразу. Память может сидеть внутри скилла, который активируется на «когда видишь задачу X — я подкачиваю уроки по X». Это встроенный в платформу routing, которого не было полгода назад.

## Жёсткие требования к финальной системе (повтор — это якоря)

1. **Auto-save через хуки** (zero ceremony). Ручные команды = провал.
2. **Self-learning** — не повторять ошибки. Главная ценность.
3. **Token economy** — lazy retrieval. Не раздувать SessionStart.
4. **Cost <50 USD/mo.** Идеал: только текущая Anthropic подписка (Claude Agent SDK headless), без доп SaaS.
5. **Transparency** — пользователь должен видеть содержимое глазами.
6. **Moat-awareness** — Anthropic скоро выкатит свою память. Быть умнее (self-learning, cross-project, compile) или нишевее.
7. **Portable** — файлы или легко экспортируется, минимум vendor lock-in.
