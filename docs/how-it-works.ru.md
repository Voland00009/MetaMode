# Как это работает

Подробный разбор пайплайна MetaMode, архитектуры и принятых проектных решений.

---

## Обзор

MetaMode имеет два входных пайплайна (захват сессий + внешний импорт) и один выходной пайплайн (компиляция wiki). Всё проходит через daily logs как промежуточный формат.

```
                   ┌──────────────────────────┐
                   │     Input Pipelines       │
                   ├──────────────────────────┤
                   │                          │
  ┌────────────┐   │   ┌──────────┐           │
  │ Session    │───┼──▶│ flush.py │──┐        │   ┌─────────────┐
  │ End/Compact│   │   └──────────┘  │        │   │  Daily Logs  │
  └────────────┘   │                 ├────────┼──▶│  daily/*.md  │
  ┌────────────┐   │   ┌──────────┐  │        │   └──────┬──────┘
  │ !save      │───┼──▶│ direct   │──┘        │          │
  └────────────┘   │   │ write    │           │          │
  ┌────────────┐   │   └──────────┘           │          ▼
  │ RAW inbox  │───┼─────────────────────────────▶┌─────────────┐
  └────────────┘   │                          │   │ compile.py  │
                   └──────────────────────────┘   └──────┬──────┘
                                                         │
                   ┌──────────────────────────┐          ▼
                   │     Output Pipeline       │   ┌─────────────┐
                   ├──────────────────────────┤   │ Wiki         │
  ┌────────────┐   │   ┌──────────────┐       │   │ knowledge/   │
  │ Next       │◀──┼───│session_start │◀──────┼───│ index.md     │
  │ Session    │   │   │.py           │       │   └─────────────┘
  └────────────┘   │   └──────────────┘       │
                   └──────────────────────────┘
```

## 1. Пайплайн захвата сессий

Это основной цикл — полностью автоматический после [настройки](setup.ru.md).

### Шаг 1: Срабатывание hook

Когда сессия Claude Code завершается (или происходит compaction контекста), hook `SessionEnd` / `PreCompact`:

1. Читает JSONL-транскрипт из файла транскриптов Claude (путь приходит через stdin)
2. Извлекает последние 30 реплик в формате markdown (пары user/assistant)
3. Обрезает до 15 000 символов по границам реплик — без обрезки на середине предложения
4. Записывает контекст во временный файл: `scripts/session-flush-<session_id>-<timestamp>.md`
5. Запускает `flush.py` как **фоновый процесс** и немедленно завершается

Сам hook не делает **ни одного API-вызова** — только локальный файловый I/O. Выполняется менее чем за 1 секунду.

**Защита от рекурсии:** если установлена переменная окружения `CLAUDE_INVOKED_BY` (означает, что Agent SDK породил этот экземпляр Claude), hook немедленно завершается. Без этого flush.py запускал бы Claude Code → который бы вызвал SessionEnd → который бы запустил flush.py → бесконечный цикл.

**Минимум реплик:** `SessionEnd` требует минимум 1 реплику. `PreCompact` требует 5 реплик — чтобы избежать сохранения крошечных compaction.

### Шаг 2: Извлечение (flush.py, проход 1)

`flush.py` читает временный файл контекста и отправляет его в Claude Agent SDK со структурированным промптом извлечения:

```
Extract from this conversation:
- Context (what the user was working on)
- Key Exchanges (important Q&A)
- Decisions Made (with rationale)
- Lessons Learned (gotchas, patterns)
- Action Items (follow-ups)
```

Если ничего стоящего для сохранения не найдено, Claude отвечает `FLUSH_OK` — маркер, который записывается в daily log, чтобы вы видели, что сессия была обработана.

### Шаг 3: Аудит качества (flush.py, проход 2)

Извлечённый контент проходит через второй вызов Agent SDK — аудит качества:

```
Is this content WORTH KEEPING in a daily log?
Mark as LOW quality only if ALL are true:
1. No concrete lessons (just "we did X, then Y")
2. No decisions with rationale
3. No reusable patterns or gotchas
4. Only routine operations
```

Аудит **консервативен** — при сомнениях контент сохраняется. Это сделано намеренно: потерять хорошую запись хуже, чем сохранить посредственную.

**Если аудит отмечает контент:** он оборачивается в HTML-комментарий:
```
<!-- AUDIT_FLAG: only routine file operations -->
Content here...
```

Отмеченные записи остаются в daily log (никогда не удаляются), но **пропускаются при компиляции**. Вы можете просмотреть и вручную снять флаг, если не согласны.

**Если аудит падает с ошибкой:** контент сохраняется как есть. Система никогда не теряет данные из-за ошибок аудита.

### Шаг 4: Запись в daily log

Результат добавляется в `daily/YYYY-MM-DD.md` с заголовком секции и временной меткой:

```markdown
### Session a1b2c3d4 (14:32)

**Context:** User was implementing auth middleware...

**Key Exchanges:**
- ...
```

Daily logs работают в режиме **только добавление** — flush никогда не модифицирует существующие записи. Это сохраняет чистоту git diff и делает компиляцию идемпотентной.

### Шаг 5: Дедупликация и автокомпиляция

- **Дедупликация:** если тот же session ID был обработан в течение последних 60 секунд, повторная обработка пропускается. Это обрабатывает случай, когда SessionEnd и PreCompact срабатывают близко друг к другу.
- **Автокомпиляция:** если сейчас после 18:00 (настраивается через `COMPILE_AFTER_HOUR`) и daily log сегодняшнего дня изменился с момента последней компиляции, `flush.py` запускает `compile.py` в фоне.

## 2. Пайплайн компиляции

Превращает daily logs в структурированные wiki-статьи.

### Как работает compile.py

1. **Поиск нескомпилированных логов:** сравнивает хеши файлов с `state.json` — обрабатывает только логи, изменившиеся с последней компиляции
2. **Чтение контекста:** загружает daily log + `AGENTS.md` (схема wiki-статьи) + существующий wiki index
3. **Отправка в Agent SDK:** Claude получает файловые инструменты (Read, Write, Edit, Glob, Grep) и до 30 реплик для создания/обновления статей
4. **Результат:** создаёт или обновляет файлы в `knowledge/concepts/`, `knowledge/connections/`, обновляет `knowledge/index.md` и `knowledge/log.md`
5. **Отслеживание состояния:** сохраняет хеши файлов в `state.json`, чтобы неизменённые логи не перекомпилировались

### Что создаёт Agent

- **Concept-статьи** — атомарные знания с YAML frontmatter (title, aliases, tags, category, sources, dates), структурированные как Context → Problem → Lesson
- **Connection-статьи** — связи между концепциями, показывающие, как две или более идеи соотносятся друг с другом
- **Обновления index** — одна строка на статью в `knowledge/index.md`
- **Записи changelog** — операции с временными метками в `knowledge/log.md`

### Контент, отмеченный аудитом

Записи, обёрнутые в `<!-- AUDIT_FLAG: ... -->`, невидимы для компилятора. Он обрабатывает только чистый, неотмеченный контент.

## 3. Внешний импорт

Для контента извне сессий Claude Code — статей, заметок со встреч, исследований.

1. Поместите `.md` или `.txt` файлы в директорию `raw/`
2. Выполните `uv run python scripts/ingest_raw.py`
3. Тот же пайплайн Agent SDK, что и при компиляции — читает документ + схему + существующий wiki
4. Создаёт wiki-статьи, обновляет index
5. Перемещает обработанные файлы в `raw/processed/`

Формат RAW inbox намеренно прост:
```markdown
# Title
## Context
## Key Insight
## Example (optional)
```

## 4. Инъекция контекста

В начале каждой сессии Claude Code `session_start.py` запускается и выводит JSON, который Claude видит сразу:

```json
{
  "additionalContext": "## Knowledge Base Index\n...\n## Recent Daily Log\n..."
}
```

### Что инжектируется

1. **Полный wiki index** — вся таблица `knowledge/index.md` (каждая статья, одна строка)
2. **Недавний daily log** — последние 30 строк из сегодняшнего или вчерашнего лога
3. **Напоминание о компиляции** — если 3+ daily logs не скомпилированы или 3+ дней с последней компиляции
4. **Уведомление о RAW inbox** — если в `raw/` есть необработанные файлы
5. **Напоминание о lint** — если wiki lint не запускался 7 дней или memory lint — 14 дней

### Бюджет размера

Общий объём инжектируемого контекста ограничен **20 000 символами** (`MAX_CONTEXT_CHARS`). Если wiki index вырастает сверх этого, он обрезается. Для справки: 20K символов — это примерно 5K токенов, малая доля контекстного окна Claude.

## 5. Аудит качества — зачем он нужен

### Проблема

Ранние эксперименты с полностью автоматическим захватом памяти (аналогично [mem0](https://github.com/mem0ai/mem0)) показали, что **~95% автоматически захваченного контента было мусором** — рутинные операции, тривиальные обмены, контент без ценности для базы знаний.

Без фильтрации wiki заполняется шумом. Компиляция производит низкокачественные статьи. Инъекция контекста тратит токены на бесполезную информацию.

### Решение

Двухпроходный LLM-пайплайн:

1. **Проход 1 (Извлечение):** извлекает структурированные секции из сырого транскрипта
2. **Проход 2 (Аудит):** отдельный LLM-вызов проверяет, соответствует ли извлечённый контент минимальным порогам качества

Это паттерн [two-pass LLM quality audit](https://en.wikipedia.org/wiki/Extract,_transform,_load) — сначала извлечение, потом аудит. Второй проход ловит то, что пропустил первый, потому что работает с чистым, структурированным текстом, а не с сырым диалогом.

### Принципы проектирования

- **Консервативность:** при сомнениях — сохранить контент. Ложноотрицательные срабатывания (потеря хорошего контента) хуже ложноположительных (сохранение посредственного контента)
- **Неразрушающий подход:** отмеченный контент никогда не удаляется, только оборачивается в HTML-комментарии
- **Fail-open:** если LLM-вызов аудита падает, контент сохраняется как есть
- **Ручное переопределение:** удалите комментарий `<!-- AUDIT_FLAG: ... -->`, чтобы снять флаг с контента

## 6. Структура Wiki

### knowledge/index.md — Главная навигация

Каждая статья — одна строка:
```
| [[concepts/slug]] | One-line summary | source-file | date |
```

Claude читает этот файл первым при старте сессии, чтобы знать, что есть в wiki. Это оглавление.

### knowledge/concepts/ — Атомарные знания

Каждая статья содержит:
- **YAML frontmatter:** title, aliases, tags, category, sources, даты created/updated
- **Тело:** структура Context → Problem → Lesson
- **Секции:** Key Points, Details, Related Concepts
- **Ссылки:** wikilinks на связанные статьи (`[[concepts/other-slug]]`)

### knowledge/connections/ — Связи между концепциями

Для неочевидных связей между 2+ концепциями. Пример: «duck typing и import binding — оба являются багами пермиссивности Python, где неверные данные проходят молча».

### knowledge/qa/ — Сохранённые ответы на запросы

Создаются через `query.py --file-back`. Содержат вопрос, синтезированный ответ и статьи-источники.

### knowledge/log.md — Журнал операций

Журнал с временными метками каждой операции compile, ingest и lint. Полезен для отладки («когда эта статья последний раз обновлялась?»).

## 7. Проверки здоровья

`lint.py` выполняет 7 проверок wiki:

| # | Проверка | Что выявляет |
|---|----------|-------------|
| 1 | **Битые ссылки** | `[[concepts/foo]]`, но `foo.md` не существует |
| 2 | **Страницы-сироты** | Статья существует, но ничто на неё не ссылается |
| 3 | **Источники-сироты** | Daily log не был скомпилирован в статьи |
| 4 | **Устаревшие статьи** | Исходный daily log изменился после компиляции статьи |
| 5 | **Отсутствующие обратные ссылки** | A ссылается на B, но B не ссылается обратно на A |
| 6 | **Разреженные статьи** | Менее 200 слов — вероятно, неполная статья |
| 7 | **Противоречия** | LLM читает все статьи и находит конфликтующие утверждения |

Проверки 1–6 структурные (быстрые, без API-вызовов). Проверка 7 использует Agent SDK и дорогая — пропустите её с `--structural-only`.

Отчёты сохраняются в `reports/lint-YYYY-MM-DD.md`. Hook старта сессии напомнит вам, если lint не запускался 7 дней.

## 8. Жизненный цикл Hooks

### Конфигурация

Hooks находятся в `~/.claude/settings.json` — глобальная конфигурация, которая срабатывает для каждой сессии Claude Code, в каждом проекте. Подробную конфигурацию смотрите в [setup.ru.md](setup.ru.md).

### Последовательность выполнения

```
Claude Code session starts
  └─▶ SessionStart hook → session_start.py → JSON to stdout → Claude sees wiki context

User works normally...

Claude Code context compacts (window full)
  └─▶ PreCompact hook → pre_compact.py → extract transcript → spawn flush.py
        └─▶ flush.py (background): extract → audit → daily log

User closes session
  └─▶ SessionEnd hook → session_end.py → extract transcript → spawn flush.py
        └─▶ flush.py (background): extract → audit → daily log
              └─▶ maybe_trigger_compilation() → spawn compile.py (after 18:00)

User types "!save important note"
  └─▶ UserPromptSubmit hook → user_prompt_submit.py → write to daily log → exit(2)
       (exit code 2 = block prompt from reaching Claude)
```

### Ключевые механизмы безопасности

- **Защита от рекурсии:** переменная окружения `CLAUDE_INVOKED_BY` предотвращает срабатывание hooks, когда Agent SDK запускает Claude
- **Дедупликация flush:** тот же session ID + окно 60 секунд = повторный flush пропускается
- **Неблокирующий режим:** hooks запускают фоновые процессы и немедленно завершаются — без задержки для пользователя
- **Отказоустойчивость:** ошибки hooks логируются в `scripts/flush.log`, но никогда не показываются пользователю и не блокируют Claude Code

### Особенности Windows

- Флаг `subprocess.CREATE_NO_WINDOW` предотвращает мелькание окон консоли при запуске фоновых процессов
- `sys.stdin.reconfigure(encoding="utf-8")` в hooks — Windows по умолчанию использует кодировку cp1251/cp1252
- Парсинг JSON включает fallback для неэкранированных обратных слешей в путях Windows
