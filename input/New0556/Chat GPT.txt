## Conclusions

1. **Do not build a full automated health monitoring system right now.**
   At ~20 articles, that is almost certainly overengineering. You do not have a scale problem. You have a **discipline/trigger problem**: the lint exists, but it does not run reliably.

2. **You are solving one real problem and three mostly imagined ones.**
   The real problem is **silent breakage from automation**: bad wiki links, duplicate generated memory files, and index/config files creeping toward hard limits.
   The mostly imagined problems are **orphan detection, stale article monitoring, and general health dashboards** at this size.

3. **The correct move is not “monitoring,” it is “cheap automatic checks.”**
   Do not build a watcher, daemon, dashboard, metrics layer, or separate health subsystem. Add a tiny gate that runs existing checks automatically at the moments that matter.

4. **The laziest useful solution is enough.**
   A single script that runs:

* on session start, or
* on save/capture completion, or
* once per day when the repo is touched

and warns only on **actual damage conditions**.

## Arguments

### 1. At your current scale, the cost of infrastructure is likely higher than the cost of the failures

With ~20 articles and growth of 5–10/week, your wiki is still small enough that:

* manual inspection is still cheap,
* failures are still recoverable,
* the graph is still understandable without tooling,
* the operational surface area is tiny.

A monitoring system has its own maintenance cost:

* config
* false positives
* threshold tuning
* alert fatigue
* debugging the monitor itself
* gradual expansion into a mini-platform

That is exactly how people waste time on “wiki ops” instead of using the wiki.

### 2. The real risk is not “health”; it is “silent corruption”

Your dangerous cases are narrow:

* broken links that make retrieval/navigation fail
* duplicate auto-generated memory files that bloat or pollute recall
* index/config size drifting toward hard limits and eventually breaking prompts or loaders
* lint not being run, so small problems accumulate invisibly

Those are real because they can degrade the system’s output quality.
By contrast:

* **orphan pages** are not inherently harmful in a small personal wiki
* **stale pages** are often fine in a knowledge base unless freshness is operationally critical
* **count metrics** are vanity unless they trigger action

So the problem is real, but smaller than your proposed solution.

### 3. The threshold where automation becomes justified is not article count alone

Automation becomes genuinely valuable when one of these becomes true:

1. **Volume**: around **100–150 pages**
   At this point, manual awareness collapses.

2. **Growth**: consistently **15–25+ new/changed pages per week**
   Now silent drift compounds faster than you will remember to check.

3. **Automation share**: **30–50%+ of files are machine-generated**
   Generated content creates repeated structure failures, duplicates, naming drift, stale fragments.

4. **Multiple ingestion paths**
   If content is arriving from sessions, background compilers, imports, scripts, and manual notes, breakage becomes non-local.

5. **Hard downstream consequences**
   If an oversized index or bad config can break session startup, retrieval, prompt assembly, or agent behavior, automation is justified earlier.

So the threshold is not “I hit 40 files.”
The threshold is “manual trust is no longer enough.”

### 4. Right now, a weekly manual lint is probably sufficient — but only if it actually happens

Your current failure is obvious: the script exists but is not part of a reliable workflow.

A manual weekly run is acceptable **only if**:

* you actually do it,
* failures do not cause immediate downstream breakage,
* the wiki is not mission-critical to every session.

But manual processes decay. So the honest answer is:

* **manual weekly lint is conceptually enough**
* **human memory is not reliable enough to depend on it**

Therefore the right fix is one inch above manual: **automatic reminder or lightweight trigger**, not full monitoring.

## What you should actually build

Build exactly this, nothing more:

### Minimum viable protection

One script, one output, one trigger.

The script checks only four things:

1. broken wiki links
2. duplicate generated memory files
3. index character count vs hard limit
4. config file size drift beyond a simple threshold

And the trigger is one of:

* at session start
* after automated capture finishes
* if lint has not run in 7 days
* if more than N files changed since last run

That is enough.

### What not to build

Do **not** build:

* dashboards
* background monitoring service
* scoring systems
* trend charts
* “knowledge health” abstractions
* orphan/staleness analytics beyond a basic warning

That is fake sophistication at your scale.

## The laziest solution that still prevents actual damage

This is the version I would recommend:

1. Keep your existing lint script.
2. Save a tiny state file with:

   * last lint run timestamp
   * last checked index size
   * last checked config size
3. On session start:

   * if lint has not run in 7 days, print warning
   * if files changed since last lint, print warning
4. On automated capture completion:

   * run only fast checks on touched files
5. Fail loudly only for:

   * broken links
   * duplicate generated files
   * index size near hard cap
   * config unexpectedly large

Everything else should be informational at most.

Example policy:

* **red**: broken links, duplicate file collision, index > 90% of cap
* **yellow**: lint older than 7 days, config grew > 25% since last baseline
* **ignore**: orphans, “stale” pages, page counts

That gives you real protection with almost no system complexity.

## Limits

This answer assumes:

* local markdown wiki
* low collaboration
* recoverable failures
* no compliance or archival requirements
