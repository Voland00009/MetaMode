## Conclusions

1. Your design is already partially obsolete. Claude Code now has native auto memory that writes plain Markdown to a local per-project memory directory, loads `MEMORY.md` at session start, and exposes a `/memory` command to inspect and edit it. Claude Code also already saves local JSONL transcripts for sessions. So your custom compile-on-idle pipeline is not filling an obvious product hole; it is building a second memory system on top of a first one. ([Claude][1])

2. Your “$0/mo extra” claim is weak to the point of being misleading. The official Agent SDK path is API-oriented, with API-key setup in the quickstart and normal token pricing on the platform docs. Claude’s docs say Max/Pro subscribers have usage included for Claude Code sessions, but they do not say headless SDK automation is covered by Max. Until you verify that experimentally, assume background compile means API billing or brittle unsupported auth tricks. ([Claude API Docs][2])

3. “Self-learning” is where this collapses. Extracting “lessons learned” from raw logs is not the same as learning. Without strong filtering, contradiction handling, retrieval evaluation, and salience control, you get a memory graveyard: lots of plausible notes, low actual recall value, and periodic wrong retrievals that make Claude worse, not better. Anthropic’s own native auto memory already tries to learn from corrections and preferences, so your external compiler must beat the built-in system, not just exist beside it. ([Claude][1])

4. The “~200-line MVP” is undercooked. File I/O is easy. Reliable memory is not. The hard parts are event semantics, async timing, dedupe, staleness, partial failures, interrupted sessions, redaction, source precedence, conflict resolution with native memory, and proving recall quality with actual evals. The hooks docs alone already expose enough edge cases to blow past the tidy MVP story. ([Claude API Docs][3])

## 1. Hidden cost: where “$0/mo” dies

**Severity: Fatal**

Your hidden-cost problem has three layers.

First, the official SDK path is not “free because Max exists.” The Agent SDK quickstart tells you to install the SDK and set an API key. The API platform has explicit token pricing for Sonnet, Opus, and Haiku. The Claude Code cost docs distinguish subscriber usage from API usage, but they do not state that SDK-driven headless jobs are covered by Max. That gap is not a detail; it is the whole economics of your architecture. ([Claude API Docs][2])

Second, background compilation is exactly the kind of workload that burns tokens invisibly. Anthropic’s own cost docs say Claude Code charges by token consumption and that costs vary with model choice, codebase size, automation, multiple instances, and context size. They also note average enterprise usage around $13 per active developer day and $150–250 per month, with 90% under $30 per active day. Your single-user compiler is smaller than enterprise use, but the principle is the same: every idle compile is another agent run with its own prompt, context, and output. ([Claude API Docs][4])

Third, your optimistic math still hurts. Under current API pricing, a Sonnet 4.6 compile that reads 50k input tokens and emits 5k output tokens costs about $0.225. Run that 4 times a day and you are at about $27/month. Double the workload to 100k in and 10k out and you are at about $54/month. Haiku 4.5 is cheaper, but then you are asking a weaker model to do the most judgment-heavy part of the system: deciding what was actually a reusable lesson. That is a bad place to economize. These estimates are still optimistic because they assume simple message pricing, not extra agent turns, retries, tool chatter, or cache misses. ([Claude API Docs][5])

Prompt caching does not rescue this as much as you probably hope. Automatic caching helps repeated context, but default TTL is 5 minutes, 1-hour TTL costs more to write, and cache reads only help when repeated content is actually reused inside the cache window. An “idle compiler” that wakes up periodically over changing logs gets much worse cache leverage than an interactive session with repeated stable prefixes. ([Claude API Docs][6])

## 2. Where “self-learning” turns into theater

**Severity: Fatal**

Your system assumes the model can infer true reusable lessons from noisy raw transcripts. Usually it cannot.

Concrete failure case one: wrong causality. A session ends with “tests pass after clearing pnpm store.” The real cause was a corrupted generated client or a stale env var, but the compiler writes “when tests fail, clear pnpm store first.” That note is now false doctrine. Next session it gets surfaced, Claude follows it, and you waste time performing ritual cleanup instead of fixing the actual class of bug.

Failure case two: retrieval miss by phrasing drift. Stored lesson: “Never edit generated Prisma client.” Future prompt: “Fix the broken field mapping in `.prisma/client`.” Your trigger logic fails because the new wording does not match the old lesson’s vocabulary. Result: the note exists, but does not fire.

Failure case three: contradiction accumulation. In March the system learns “use Vitest.” In June the repo standard changes to Playwright for integration flows. Unless you have explicit contradiction detection and downgrade rules, both memories survive. The model then sees two authoritative-looking notes and hallucinates a compromise or picks the wrong one. Native memory systems like Mem0 explicitly market ADD/UPDATE/DELETE/NOOP memory operations because this is the real problem. Your design, as described, does not solve it. ([Mem0][7])

Failure case four: salience bias. Your “important moments” manual marker only captures what the user notices in real time. The expensive lessons are often the ones the user did **not** notice: hidden false starts, dead-end grep patterns, environment traps, wrong assumptions Claude made before eventually recovering. Raw transcript compile might catch some of this, but only if the compiler is good at causal analysis, which is exactly the hard part.

Failure case five: timing mismatch. Async hooks deliver context on the **next** turn and do not deduplicate executions. So even if you compile something useful in the background, it may arrive too late, arrive duplicated, or arrive after the next task has already gone off the rails. ([Claude API Docs][3])

## 3. Where the “~200-line MVP” breaks

**Severity: High**

The naïve version is 200 lines. The real version is not.

You are underestimating hook behavior. `Stop` does not run on user interrupt. API errors fire `StopFailure` instead. `UserPromptSubmit`, `Stop`, and several other events ignore matchers and fire every time. `Stop` has `stop_hook_active` specifically because bad hook logic can keep Claude running indefinitely. Async hooks spawn separate background processes with no deduplication and only deliver their output on the next turn. That is already enough complexity to require real state management, lock discipline, and loop prevention. ([Claude API Docs][3])

Then there is the boring engineering you are hand-waving away: atomic writes, crash-safe append logs, multi-session concurrency, repo/worktree identity, compaction-aware parsing, retry policy, log rotation, secret redaction, schema evolution, stale index rebuilds, memory corruption recovery, and explicit precedence between native `MEMORY.md` and your custom lesson files. Claude Code itself is changing quickly enough that the release notes already show fixes around settings merges, SDK changes, lingering processes, hanging exit behavior, and shell configuration issues. Building on top of a moving target while pretending this is “just markdown files” is unserious. ([Claude API Docs][3])

The ugliest hidden corner is coexistence with native auto memory. Claude Code already stores project memory in `~/.claude/projects/<project>/memory/`, already uses `MEMORY.md` as the session-start entrypoint, already lets Claude write topic files, and already loads the first 200 lines or 25KB at startup. If you inject a second parallel index and a second parallel lesson format, you now own reconciliation forever. ([Claude][1])

## 4. Assumptions that collapse on real data

**Severity: High**

You are assuming disk bloat is the risk. Wrong. Retrieval bloat is the risk. Claude Code already writes each message, tool use, and result to local JSONL. Over months, the real problem is not that storage gets large; it is that the event stream is mostly garbage for memory purposes. Too much low-signal raw material makes your compiler produce bland summaries and duplicate “lessons.” ([Claude][8])

You are assuming a tiny session-start index remains stable. It does not. Small indexes rot fast because they become a dumping ground for “important” items with no expiry model. Anthropic’s own design solves this by treating `MEMORY.md` as a concise entrypoint and moving detail into topic files, while hard-capping what loads at startup. Your design is converging toward the exact same structure, except with extra machinery and no proof it will manage staleness better. ([Claude][1])

You are assuming project boundaries are obvious. They are not. Native auto memory is repo-derived and shared across worktrees and subdirectories within the same git repo. That is convenient until one mono-repo contains unrelated services, environments, or experiments. Then memory bleed becomes real. Your design needs a project segmentation story that is better than “folder plus markdown,” or you will contaminate lessons across domains. ([Claude][1])

You are assuming prompt/compiler drift is manageable. It is not unless you pin evaluation. The moment you tweak compile prompts, schema, or lesson taxonomy, old memories become semantically inconsistent with new ones. That is schema migration, even if you pretend Markdown makes it informal.

## 6. Contradictions in your 7 requirements

**Severity: Fatal**

Your requirements are not jointly satisfiable in the strong form.

## 9. `[MEM]` inside chat is bad UX and bad systems design

**Severity: High**

`[MEM]` is sloppy.

It collides with normal content. It can appear in code, markdown, test fixtures, docs, logs, or literal user text. It pollutes the prompt unless stripped. If stripped, you need mutation logic that must preserve the actual user meaning. On mobile or unreliable Russian input it is still typing ceremony, just uglier ceremony. It is also ambiguous whether `[MEM]` means “save the next sentence,” “save the whole prompt,” or “mark salience for the current session.” That ambiguity becomes bugs.

You have better options.

Best option: make memory marking a **dedicated slash command**, not inline text. Claude Code skills/custom commands already support `/name` invocation and load lazily, so `/mem this repo needs redis before API tests` is cleaner, collision-free, and semantically explicit. Better still, make `/mem` write to a deterministic inbox file or JSONL queue, not directly to long-term memory. Even better than chat tagging: use a **local hotkey helper** outside the chat stream. One global shortcut writes selected text or clipboard plus cwd/repo metadata into `memory-inbox.jsonl`.
